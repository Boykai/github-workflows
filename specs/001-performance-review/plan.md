## Summary

Deliver a balanced first-pass performance improvement across both the Python/FastAPI backend and the React/TypeScript frontend. The work is measurement-gated: capture trusted baselines before changing behavior, then fix the highest-value issues already surfaced — backend GitHub API churn around board refreshes and polling, and frontend board responsiveness issues caused by broad query invalidation and hot event listeners. Structural rewrites (virtualization, service decomposition) are explicitly deferred unless the first pass fails its measured targets.

## Technical Context

**Language/Version**: Python 3.12+ (backend), TypeScript 5.9 / React 19 (frontend)
**Primary Dependencies**: FastAPI ≥0.135, TanStack React Query 5.90, @dnd-kit, Vite 7.3, websockets 16
**Storage**: SQLite via aiosqlite (session/settings); in-memory TTL cache (`backend/src/services/cache.py`)
**Testing**: pytest + pytest-asyncio (backend), Vitest + @testing-library/react (frontend)
**Target Platform**: Web application (browser SPA + API server)
**Project Type**: Web — separate `backend/` and `frontend/` directories
**Performance Goals**: 0 repeated unchanged board refreshes during 10-min idle; ≥30% upstream request reduction for warm-state board refresh; 95th-percentile interaction ≤200 ms for in-scope board/chat interactions
**Constraints**: No new runtime dependencies; no board virtualization in first pass; keep existing user-facing behavior intact
**Scale/Scope**: Representative board with typical card count; improvements target steady-state board viewing, not synthetic stress
