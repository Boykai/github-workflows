# Implementation Plan: Chat Agent Auto-Generate Full GitHub Issue Metadata

**Branch**: `018-issue-metadata-autogen` | **Date**: 2026-03-05 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/018-issue-metadata-autogen/spec.md`

## Summary

Extend the existing chat agent issue-creation pipeline so that all metadata fields (priority, size, estimate, start/target dates, labels, assignees, development branch) are populated from **real** repository data rather than hardcoded constants. A new `MetadataService` fetches labels, branches, milestones, and collaborators from the GitHub REST/GraphQL APIs, persists them in a SQLite `github_metadata_cache` table with TTL-based invalidation, and injects them as structured context into the AI prompt. The frontend `IssueRecommendationPreview` gains inline-editable fields for every metadata attribute, and the `create_issue` REST call expands to include milestone, assignees, and the full labels array selected by the AI.

## Technical Context

**Language/Version**: Python 3.11 (backend), TypeScript 5.x (frontend)
**Primary Dependencies**: FastAPI ≥ 0.109, Pydantic ≥ 2.5, aiosqlite ≥ 0.20, httpx ≥ 0.26, React 18 + Tailwind CSS (frontend)
**Storage**: SQLite (`/app/data/settings.db`) — existing database with custom migration runner
**Testing**: pytest (backend), vitest (frontend), playwright (e2e)
**Target Platform**: Linux container (Docker Compose: backend:8000 + frontend:5173/80)
**Project Type**: Web application (backend + frontend)
**Performance Goals**: Issue creation completes in < 60 seconds end-to-end; cached metadata lookups < 50 ms; initial metadata fetch < 10 s even for repos with 200+ labels
**Constraints**: Must work within GitHub API rate limits (5000 req/hr authenticated); SQLite single-writer; metadata cache must survive app restarts
**Scale/Scope**: Single-user desktop app with one active repository at a time; cache keyed by `owner/repo` to support multi-repo sessions
