# Implementation Plan: Tools Page Audit

**Branch**: `043-tools-page-audit` | **Date**: 2026-03-16 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/043-tools-page-audit/spec.md`

## Summary

Comprehensive audit of the Tools page to bring it to modern best practices across reliability, accessibility, UX polish, responsive design, and code quality. The page is already well-structured at 84 lines (ToolsPage.tsx) with a clean delegation to ToolsPanel (399 lines) and 8 sub-components. The primary work targets: (1) adding missing loading/error/empty states and confirmation dialogs for all destructive actions, (2) improving keyboard navigation and ARIA compliance for screen readers, (3) polishing copy, button labels, timestamps, and dark mode consistency, (4) ensuring responsive layout from 768px to 1920px, and (5) achieving full type safety, zero lint warnings, and meaningful test coverage for hooks and key interactive components.

**Approach**: Phase-ordered implementation — Phase 1 (reliability: states + error handling) is blocking; Phase 2 (accessibility) and Phase 3 (UX polish) can run in parallel; Phase 4 (responsive/performance) follows; Phase 5 (code quality + testing) is the final cleanup pass.

## Key Decisions

| Decision | Chosen | Rejected Alternative | Rationale |
|----------|--------|---------------------|-----------|
| Delete confirmation | Use existing `<ConfirmationDialog>` from `src/components/ui/` | Keep inline confirmation markup in ToolsPanel | ConfirmationDialog is the established pattern across the app; provides consistent UX and proper focus management |
| Loading indicators | Use `<CelestialLoader>` component | Keep plain text "Loading…" messages | CelestialLoader is the standard loading component with proper sizing and animation |
| Error message format | "Could not [action]. [Reason]. [Next step]." | Raw error messages from API | User-friendly format matches spec requirement (FR-008) and is consistent with other pages |
| Timestamp display | Relative time for recent (<24h), absolute for older | Always relative or always absolute | Mixed approach provides best UX — recent events show "2 hours ago", older events show full date |
| Tooltip for truncation | Wrap truncated text in `<Tooltip>` component | CSS-only title attribute | Tooltip component provides consistent styling and proper accessibility |
| Test scope | Hook tests + key component interaction tests | Full snapshot tests | Assertion-based tests are more maintainable; spec explicitly excludes snapshot tests |
