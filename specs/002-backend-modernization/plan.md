# Implementation Plan: Backend Modernization & Improvement

**Branch**: `002-backend-modernization` | **Date**: 2026-03-15 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/002-backend-modernization/spec.md`

## Summary

Modernize the Solune FastAPI backend by addressing 12+ anti-patterns causing resource leaks, data inconsistency risks, and silent failures. The work spans 5 phases (22 functional requirements across 20 steps): async safety (TaskGroup, TaskRegistry, timeouts, WebSocket resilience), data integrity (single source of truth, transaction boundaries, race condition prevention), security hardening (CSRF, indexes, scoped cache keys, compound rate-limit keys), performance optimization (filtered queries, pagination, reduced TTL, safe task eviction), and modern Python patterns (enums, protocols, TypedDict, consistent error handling, thread offloading).

**Approach**: Phase-ordered implementation ã Phase 1 (async safety) must complete before Phases 2ã3 (which can run in parallel), Phase 4 follows, and Phase 5 is independent cleanup.

## Implementation Phases

### Phase 1: Critical Async Safety *(blocks Phases 2-5)*

#### Task 1.1: Adopt `asyncio.TaskGroup` for Startup Background Tasks

**FR**: FR-001 | **Files**: `main.py:370-440`
**Change**: Replace manual `asyncio.create_task()` + `cancel()` pattern in `lifespan()` with Python 3.11+ `asyncio.TaskGroup`. The task group context manager automatically cancels and awaits all child tasks when the `async with` block exits.
**Affected tasks**: `_session_cleanup_loop()`, `_polling_watchdog_loop()`, Signal WebSocket listener, agent MCP sync.
**Note**: Signal bridge `start_signal_ws_listener()` (line 477) also uses `create_task` ã integrate into the task group or delegate to TaskRegistry (Task 1.2).

#### Task 1.2: Build `TaskRegistry` for Fire-and-Forget Tasks

**FR**: FR-002 | **New file**: `services/task_registry.py`
**Change**: Create a singleton `TaskRegistry` class that:
- Wraps `asyncio.create_task()` with automatic registration
- Tracks task state (pending, running, completed, cancelled, failed)
- Provides `drain(timeout)` method for graceful shutdown
- Logs tasks that fail to complete within drain timeout
**Affected call sites** (12+ locations):
- `api/chat.py:681` ã Signal delivery
- `api/signal.py` ã Signal post link
- `services/signal_bridge.py:667` ã AI processing
- `services/signal_delivery.py:279` ã Delivery task
- `services/copilot_polling/__init__.py:285` ã Poll task
- `services/github_projects/service.py:328` ã GraphQL execution
- `services/model_fetcher.py:323` ã Background refresh
- `services/workflow_orchestrator/transitions.py` ã Transition coroutines
- `main.py:401` ã Agent MCP sync
**Post-implementation**: Remove `RUF006` suppression from `pyproject.toml:70`; run `ruff check --select=RUF006` to verify zero violations.

#### Task 1.3: Add Timeout Guards on External API Calls

**FR**: FR-003 | **Files**: `services/github_projects/service.py:324-336`
**Change**: Wrap `client.async_graphql()` with `asyncio.wait_for(coro, timeout=30)` or configure httpx timeout on the client. Also audit other external HTTP calls (model fetcher, copilot polling) for missing timeouts.
**Timeout strategy**: 30s default for GitHub GraphQL; configurable via `Settings` for operator tuning.

#### Task 1.4: Fix Signal WebSocket Resource Leak

**FR**: FR-004 | **Files**: `services/signal_bridge.py:509-530`
**Change**:
- Current: `_process_inbound_ws_message()` exceptions (other than `TimeoutError`) are caught but the WebSocket context manager handles cleanup. The outer `except Exception` block reconnects after a flat 10s sleep.
- Fix: Add exponential backoff with jitter for reconnection (base=1s, cap=300s). Reset backoff on successful connection. Ensure `finally` block explicitly closes WebSocket if context manager fails.
- Add configurable max reconnect delay.

### Phase 2: Data Integrity & Persistence *(parallel with Phase 3)*

#### Task 2.1: Eliminate Chat Dual Source of Truth

**FR**: FR-005 | **Files**: `api/chat.py:91-93, 96-152`
**Change**: Refactor `_messages`, `_proposals`, `_recommendations` from primary stores to read-through caches:
- All writes go to SQLite first (via `chat_store`)
- On write success, update in-memory cache
- On cache miss, load from SQLite
- Protect cache updates with `asyncio.Lock` to prevent concurrent modification
**Note**: The current code comments (lines 78-93) already describe this as "write-through cache" but the implementation treats memory as primary. Align implementation with documented intent.

#### Task 2.2: Fix Fire-and-Forget Persistence Silent Failures

**FR**: FR-006 | **Files**: `api/chat.py:96-152`
**Change**: Replace bare `except Exception` + `logger.warning()` with:
- Retry mechanism (3 attempts with exponential backoff) for transient failures (database locked, I/O errors)
- Propagate persistent failures to caller via exception
- Add structured logging with failure context (session_id, message_id, attempt number)

#### Task 2.3: Add Database Transaction Boundaries

**FR**: FR-007 | **Files**: `services/chat_store.py`
**Change**: Wrap multi-step operations in explicit transactions:
- Use `async with db.execute("BEGIN IMMEDIATE")` for write transactions
- Add savepoints for nested operations
- Rollback on any step failure
- Remove per-operation `await db.commit()` (transaction handles commit)

#### Task 2.4: Fix Admin Auto-Promotion Race Condition

**FR**: FR-008 | **Files**: `dependencies.py:132-150`
**Change**: The current code already uses `WHERE admin_github_user_id IS NULL` (line 141) which is atomic in SQLite. However:
- Verify `cursor.rowcount` is checked correctly after the conditional UPDATE
- Add explicit handling for the "lost race" case (rowcount == 0) ã re-read the admin user and return 403 if the promoted user is different
- Add integration test: 10 concurrent requests, exactly 1 succeeds

## Key Decisions

| Decision | Rationale | Alternatives Rejected |
|----------|-----------|----------------------|
| `asyncio.TaskGroup` for startup tasks | Built-in Python 3.11+, automatic cancellation and awaiting, no external dependency | Manual cancel/await pattern (current ã error-prone, leaks) |
| Singleton `TaskRegistry` | Single point of control for 12+ scattered `create_task()` calls; enables drain-on-shutdown | Per-module task tracking (fragmented, hard to drain); keep RUF006 suppression (hides real bugs) |
| Double-submit cookie CSRF | Stateless, works with cookie-based sessions, no server-side token storage needed | Synchronizer token (requires server state); SameSite cookie only (insufficient for subdomain attacks) |
| SQLite `BEGIN IMMEDIATE` transactions | Prevents writer starvation in WAL mode; compatible with aiosqlite | `BEGIN DEFERRED` (allows readerãwriter promotion failure); `BEGIN EXCLUSIVE` (too aggressive for reads) |
| Compound rate-limit key (user ID + IP) | Prevents bypass via cookie clearing; maintains per-user fairness | Session-only key (current ã bypassable); IP-only (unfair for shared NATs) |
| 5-minute metadata cache TTL | Balances freshness (branches change frequently) with API rate limits | 1-hour (current ã too stale); webhook invalidation (complex, deferred as enhancement) |
| `StrEnum` for string constants | Drop-in replacement (StrEnum is also a str), backward compatible with existing string comparisons | Regular Enum (breaks string comparisons); no change (typo-prone) |
